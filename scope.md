# Scope

## Project Title

**Rule-driven Trace Canonicalization for Runtime Verification of LLM Agents**

## Goal

Build and evaluate a prototype that converts heterogeneous LLM-agent raw traces into a normalized event stream and checks whether the normalized stream preserves the evidence required by verification rules.

The goal is not to build a new verifier from scratch. The goal is to build a reusable canonicalization and preservation-checking layer between agent runtimes and verifier backends.

## In Scope

### 1. Supported Trace Sources

The prototype will support three trace-source types.

#### 1.1 LangChain-style trace

Traces containing events such as:

- Planner step
- Tool invocation
- Tool output
- Retrieval
- Agent message
- Final response
- Error or exception

Expected examples:

```text
Agent plan → retrieval → tool call → tool result → final answer
```

#### 1.2 MCP / tool-call trace

Traces containing events from MCP servers or structured tool execution.

Expected examples:

```text
mcp.gmail.send
mcp.calendar.create_event
mcp.files.read
mcp.memory.write
external_api.request
```

#### 1.3 Custom ReAct trace

Simple custom traces generated by a ReAct-style loop.

Expected examples:

```text
Thought → Action → Observation → Thought → Final Answer
```

This source is used for controlled synthetic evaluation and oracle labeling.

## 5. Supported Rule Classes

The prototype will support five rule classes.

### 5.1 Permission rules

Rules checking whether an action is allowed.

Example:

```text
The agent must not send an external email unless the action is allowed by policy.
```

Required evidence may include:

```text
action_name
effect_type
target_resource
allowed_action_set
approval.status
```

### 5.2 Temporal rules

Rules checking whether events occur in the correct order.

Example:

```text
Approval must occur before external send.
```

Required evidence may include:

```text
event order
step_id
parent_event
approval event
target binding
```

### 5.3 Provenance rules

Rules checking whether an output, claim, or action is grounded in a valid source.

Example:

```text
A final answer that cites retrieved content must be linked to a retrieval event.
```

Required evidence may include:

```text
input_refs
output_ref
provenance
source_id
causal_link
```

### 5.4 Taint rules

Rules checking whether untrusted data flows into a dangerous action.

Example:

```text
Data from untrusted retrieval must not flow into a destructive tool call.
```

Required evidence may include:

```text
taint
input_refs
causal_path
effect_type
target_resource
```

### 5.5 Recovery / Governance rules

Rules checking how the agent behaves after failure, blocking, rewriting, or re-planning.

Example:

```text
A failed tool call must not be reported as a successful final result.
```

Required evidence may include:

```text
status
error_type
tool output
final response link
parent_event
reversibility
decision
```

## 6. Supported Verdicts

The system will use four verdicts.

### 6.1 `SAFE`

The rule is fully evaluable and no violation is found.

Conditions:

```text
required evidence is complete
rule condition is checked
no violation is detected
```

### 6.2 `VIOLATION`

The rule is fully evaluable and a violation is found.

Conditions:

```text
required evidence is complete
rule condition is checked
policy is violated
```

### 6.3 `UNKNOWN`

The rule cannot be evaluated safely because required evidence is missing.

Conditions:

```text
required evidence is incomplete
missing evidence prevents sound evaluation
system must not report SAFE
```

Examples:

```text
missing approval status
missing temporal order
missing taint label
missing causal link
missing tool status
```

### 6.4 `INCONSISTENT`

The trace contains contradictory evidence.

Conditions:

```text
evidence exists
but evidence conflicts across events or fields
```

Examples:

```text
same event has status success and failed
approval target does not match action target but is marked same_target
causal link points to nonexistent event
```

## 7. Initial Policies for MVP

The first prototype should implement three policies.

### P001 — Approval before external send

```text
An agent must not send an external email unless there is a prior approval event for the same target.
```

Rule class:

```text
Permission + Temporal
```

Required evidence:

```text
effect_type
target_resource
approval.status
approval.target
temporal_order
causal_link
```

### P002 — No untrusted retrieval to destructive tool

```text
Data from untrusted retrieval must not flow into delete, write, send, or execute actions without an explicit approval or sanitization step.
```

Rule class:

```text
Taint + Provenance + Permission
```

Required evidence:

```text
taint
provenance
input_refs
causal_path
effect_type
target_resource
approval.status
```

### P003 — Tool failure must not be reported as success

```text
If a tool call fails, the final response must not claim that the tool action succeeded.
```

Rule class:

```text
Recovery / Governance + Provenance
```

Required evidence:

```text
tool status
error_type
final response link
parent_event
output_ref
causal_link
```

## 8. Baselines

The evaluation will compare against:

### B1 — Raw trace + custom parser per verifier

Each verifier or policy reads the raw trace using its own parser.

### B2 — Tool-call-only abstraction

Only tool name, arguments, and output are preserved.

### B3 — One existing schema only

Use a single existing event or trace format without rule-driven evidence extensions.

### B4 — Generic observability / span trace

Use span/log-style traces that are useful for debugging but not designed for verification evidence.

### Ours

Rule-driven normalized profile with evidence-preservation checking.

## 10. Out of Scope

This project will not attempt to:

1. Build a new full verifier from scratch.
2. Replace existing verifier backends such as AgentSpec, LTL, SMT, Hoare contracts, or PRISM.
3. Guarantee safety for all LLM-agent behaviors.
4. Verify policies when the raw runtime does not log required evidence.
5. Detect hidden side effects outside the trace.
6. Prove that all human-written policies are correct.
7. Support every possible agent framework.
8. Support arbitrary natural-language policies without translation into Rule IR.
9. Solve alignment or hallucination broadly.
10. Provide production-grade enforcement in the first prototype.

## 11. MVP Implementation Boundary

The MVP will be considered complete when it has:

1. A normalized event schema.
2. At least one custom ReAct trace adapter.
3. At least one LangChain-style or MCP-style adapter.
4. Three implemented policies:
   - P001 Approval before external send
   - P002 No untrusted retrieval to destructive tool
   - P003 Tool failure must not be reported as success
5. A Rule IR representation.
6. An evidence-preservation checker.
7. A simple Python verifier backend.
8. Oracle labels for synthetic traces.
9. Metrics for:
   - Evidence completeness
   - Verdict preservation
   - False safe rate
   - Unknown rate
10. One ablation experiment.

## 12. Success Criteria

The project is successful if the prototype demonstrates that:

- Raw traces from at least two formats can be normalized.
- The normalized profile supports multiple rule classes.
- Missing evidence produces `UNKNOWN`, not `SAFE`.
- False safe rate is lower than tool-call-only abstraction.
- Ablation shows that provenance, taint, causal links, and status fields matter.
- At least two verifier-style backends or backend formats can reuse the same normalized event stream.

## 13. One-sentence Scope Summary

This project studies a rule-driven canonicalization and evidence-preservation layer for LLM-agent runtime verification, focusing on whether normalized traces preserve enough information for verifier backends to avoid false-safe verdicts.
